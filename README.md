# 🏋️ Weighted Pull-Up Calculator

A web app to estimate your one-rep-max (1RM) for weighted pull-ups and track your progress over time.

**Live at:** [pullup-calculator.onrender.com](https://pullup-calculator.onrender.com)

---

## What It Does

- **Calculator tab** (public) — Enter your bodyweight and see a heatmap and line charts estimating your 1RM across different added weights and rep ranges. Uses the modified Epley formula.
- **Timeline tab** (password-protected) — Log your workouts over time and see your 1RM and estimated unweighted rep max charted over time.

## How It's Hosted — The Architecture

This app uses three free services that each handle a different job:

```
┌──────────────┐      ┌──────────────┐      ┌──────────────┐
│    GitHub     │      │    Render    │      │     Neon      │
│              │ ───► │              │ ───► │              │
│  Source code  │      │  Runs the    │      │  PostgreSQL   │
│  repository   │      │  Flask app   │      │  database     │
└──────────────┘      └──────────────┘      └──────────────┘
```

### GitHub — Code Storage
- Hosts the source code at [github.com/jameshclegg/pullup-calculator](https://github.com/jameshclegg/pullup-calculator)
- When you push changes, Render automatically detects them and redeploys

### Render — Web Hosting
- Runs the Python/Flask application and serves it to the internet
- Reads the `render.yaml` file to know how to build and start the app
- Provides the public URL (e.g. `pullup-calculator.onrender.com`)
- On the free tier, the app "sleeps" after 15 minutes of inactivity — the first visit after that takes a few seconds to wake up

### Neon — Database
- Provides a free PostgreSQL database to store timeline entries
- Without this, any data added on the live site would be lost every time Render redeploys (because Render's free tier has an ephemeral filesystem)
- On the free tier, the database scales to zero after 5 minutes of inactivity — the first query takes ~1 second to reconnect

## Environment Variables

These are set in the Render dashboard (not in code):

| Variable | Purpose |
|---|---|
| `DATABASE_URL` | Neon PostgreSQL connection string |
| `SECRET_KEY` | Flask session encryption key (auto-generated by Render) |
| `TIMELINE_PASSWORD` | Password to access the Timeline tab |

---

## Running Locally

The app works locally without a database — it falls back to storing timeline data in `data/timeline.json`.

```bash
# Install dependencies
uv sync

# Run the dev server
uv run python app.py
```

Then open [http://localhost:5050](http://localhost:5050).

To test with a database locally, set the `DATABASE_URL` environment variable to your Neon connection string before running.

## Project Structure

```

calculator.py       # 1RM calculation logic
db.py               # Database layer (PostgreSQL + local JSON fallback)
render.yaml         # Render deployment configuration
pyproject.toml      # Python dependencies (managed with uv)
templates/
  index.html        # Main app template (calculator + timeline tabs)
  login.html        # Login page for timeline access
data/
  timeline.json     # Local timeline data (used when no database)
```

---

## Key Files Explained

- **`render.yaml`** — Tells Render how to build (`uv sync`) and start (`gunicorn`) the app, and which environment variables are needed.
- **`db.py`** — Abstracts storage so the app works both locally (JSON file) and in production (PostgreSQL). Checks if `DATABASE_URL` is set to decide which to use.
- **`app.py`** — The main application. The `@login_required` decorator on the timeline route checks for a session cookie and redirects to `/login` if the user hasn't authenticated.
